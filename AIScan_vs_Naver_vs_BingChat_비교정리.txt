
[AIScan vs. Naver vs. BingChat – LLM 구조 전략 비교표]

■ 1. 설계 철학

AIScan:
- LLM이 중심이 아니라 "검색"이 중심
- 검색 데이터를 정제 → 구조화 → 감정형 LLM 응답
- Hallucination 제거 + 실제성과 감정 몰입 동시 설계

Naver:
- 자사 검색엔진 결과 위에 LLM 설명 덧붙이는 방식
- 검색 결과 → 후처리 요약 또는 부연 설명
- 아직 구조적 통합보단 부가 기능 수준

BingChat:
- GPT를 기반으로, 검색을 보조적 정보 출처로 활용
- LLM 우선 → 검색은 신뢰 보완용
- 구조는 강력하나, 한국어/현지화 제한 있음

------------------------------------------------------------

■ 2. 구조 흐름 비교

AIScan:
[사용자 질의]
    ↓
[네이버 검색 API 호출 (검색/지식인/지도 등)]
    ↓
[데이터 구조화 + 요약]
    ↓
[LLM 입력 + 감정형 UX 처리]
    ↓
[최종 응답 생성 및 출력]

Naver:
[사용자 질의]
    ↓
[네이버 검색 결과 출력]
    ↓
[LLM 부연 설명 또는 요약]
    ↓
[출력]

BingChat:
[사용자 질의]
    ↓
[GPT 모델 응답 생성]
    ↓
[실시간 웹 검색 보완 → 링크 출력]
    ↓
[LLM 응답에 참고 자료 포함]

------------------------------------------------------------

■ 3. 차별 포인트 요약

| 항목                 | AIScan                          | Naver                          | BingChat                         |
|----------------------|---------------------------------|---------------------------  ---|----------------------------------|
| 구조 주도            | 검색 → LLM                      | 검색 + LLM 덧붙임              | LLM 우선 + 검색 보완              |
| 네이티브 자산 활용   | 네이버 검색/지식인/지도 통합     | 검색결과 기반 제한적           | Bing 검색 연동                    |
| UX 설계              | 감정형 몰입 UX 포함             | 단순 요약 중심                 | 링크 및 부연 응답                 |
| Hallucination 대응   | 구조적 차단 구조 설계           | 부분 보완 (LLM 후처리)         | 검색 기반 보완 링크 삽입          |
| 한국어 대응력        | 최고 (자산 활용 기반)           | 매우 높음                      | 낮음 (다국어 기반, 한계 있음)     |
